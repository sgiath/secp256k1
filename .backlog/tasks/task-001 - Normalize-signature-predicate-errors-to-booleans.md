## Description

<!-- SECTION:DESCRIPTION:BEGIN -->
> *This was generated by AI during triage.*

Finding: `Secp256k1.ecdsa_valid?/3` and `Secp256k1.schnorr_valid?/3` are documented and typed as boolean predicates, but NIF parser failures can return `{:error, reason}`. In Elixir that tuple is truthy, so callers using these APIs directly in `if` or guard-like authorization logic can accidentally accept malformed signature or pubkey inputs.

Evidence checked:
- `lib/secp256k1.ex` specs for `ecdsa_valid?/3` and `schnorr_valid?/3` promise `boolean()`.
- `c_src/ecdsa.c` returns `error_result` when compact signature or pubkey parsing fails.
- `c_src/schnorrsig.c` returns `error_result` when x-only pubkey parsing fails.
- Focused probe returned `{:error, "secp256k1_ecdsa_signature_parse_compact failed"}` and `{:error, "secp256k1_xonly_pubkey_parse failed"}` from predicate calls.

Expected direction:
- Make predicate APIs return only `true` or `false` for all invalid cryptographic inputs.
- If error details are useful, expose them through a non-predicate API or internal helper.
- Add tests for malformed signatures and pubkeys asserting strict `false` return values.

## Implementation Brief

Root cause: the `valid?/3` NIFs are predicates but return `error_result/2` on cryptographic parser failures. In Elixir, `{:error, reason}` is truthy, so callers using `if Secp256k1.ecdsa_valid?(...)` or `if Secp256k1.schnorr_valid?(...)` can accept malformed inputs.

Change the C verify NIFs, not only the public `Secp256k1` delegates, so direct callers of `Secp256k1.ECDSA.valid?/3` and `Secp256k1.Schnorr.valid?/3` are fixed too.

Concrete edits:
- In `c_src/ecdsa.c`, make `verify` return the atom `false` when `secp256k1_ecdsa_signature_parse_compact` fails.
- In `c_src/ecdsa.c`, make `verify` return the atom `false` when `secp256k1_ec_pubkey_parse` fails.
- In `c_src/schnorrsig.c`, make `verify` return the atom `false` when `secp256k1_xonly_pubkey_parse` fails.
- Keep successful verification returning `true` and failed verification returning `false` as today.
- Keep wrong type or wrong byte-size argument handling as `badarg` unless the maintainer explicitly decides predicates should also normalize contract violations.

Do not add a non-predicate error-returning API for this ticket unless implementation uncovers a concrete caller need. The root-cause fix is preserving predicate truth semantics.
