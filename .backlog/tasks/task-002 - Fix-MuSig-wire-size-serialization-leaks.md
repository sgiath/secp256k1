## Description

<!-- SECTION:DESCRIPTION:BEGIN -->
> *This was generated by AI during triage.*

Finding: MuSig NIFs allocate and return struct-sized binaries while libsecp256k1 serializers write smaller fixed wire formats. `nonce_gen/5` returns a 132-byte pubnonce even though the serializer writes 66 bytes. `nonce_agg/1` returns a 132-byte aggnonce even though the serializer writes 66 bytes. `partial_sign/4` returns a 36-byte partial signature even though the serializer writes 32 bytes. The unwritten tail can contain uninitialized BEAM/NIF memory.

Evidence checked:
- `c_src/musig.c` allocates `sizeof(pubnonce)`, `sizeof(aggnonce)`, and `sizeof(partial_sig)` before calling serializers.
- `c_src/secp256k1/include/secp256k1_musig.h` documents 66-byte pubnonce/aggnonce serialization and 32-byte partial signature serialization.
- Focused probe observed nonzero tail bytes including BEAM path-like data after serialized MuSig outputs.
- `lib/secp256k1/musig.ex` type docs currently mix serialized and opaque struct sizes.

Expected direction:
- Allocate exact serialized wire sizes: 66 bytes for pubnonce, 66 bytes for aggnonce, 32 bytes for partial signatures.
- Enforce exact input lengths before parse calls.
- Update Elixir types/docs and tests to match wire sizes.
- Add regression tests proving returned binaries have no extra tail bytes.
