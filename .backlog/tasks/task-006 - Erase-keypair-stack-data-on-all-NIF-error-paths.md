## Description

<!-- SECTION:DESCRIPTION:BEGIN -->
> *This was generated by AI during triage.*

Finding: some NIFs create sensitive `secp256k1_keypair` stack values and return on downstream failures before calling `secure_erase`. The most likely affected paths are rare libsecp256k1 serialization/signing failures after keypair creation, but cryptographic NIFs should keep cleanup unconditional.

Evidence checked:
- `c_src/schnorrsig.c` creates a `secp256k1_keypair` in `sign32` and `sign_custom`; downstream error paths return before the final `secure_erase` call.
- `c_src/extrakeys.c` creates a `secp256k1_keypair`; downstream pubkey extraction or serialization failures return before the final `secure_erase` call.
- Existing project guidance says sensitive data should be securely erased before returning.

Expected direction:
- Refactor NIF functions that create keypair/seckey-derived stack structures to use cleanup labels or equivalent single-exit cleanup.
- Ensure every path after keypair creation calls `secure_erase` before returning.
- Preserve existing successful return behavior and error tuple behavior.
- Add focused tests where feasible, or document why these rare downstream failures are only covered by code inspection.
